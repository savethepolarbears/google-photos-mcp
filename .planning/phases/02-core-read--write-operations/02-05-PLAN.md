<objective>
Implement `upload_media` tool to satisfy WRIT-02 — two-step raw byte upload followed by batchCreate finalization.

Purpose: Google Photos upload is a two-step process. This plan delivers it as a single atomic tool call: read file bytes, POST to /v1/uploads to get uploadToken, POST to /v1/mediaItems:batchCreate to create the media item. The tool optionally accepts an albumId to place the item in an album immediately.
Output: Working `upload_media` MCP tool backed by new type definitions, client methods, repository functions, Zod schema, and handler.
</objective>

2. Add `batchCreate` to the existing `mediaItems` object, after the `get` method:
```typescript
batchCreate: async (params: {
  albumId?: string;
  newMediaItems: Array<{ uploadToken: string; fileName?: string; description?: string }>;
}) => {
  try {
    const headers = await getAuthorizedHeaders(auth);
    const response = await photosApi.post<BatchCreateResponse>(
      '/mediaItems:batchCreate',
      {
        albumId: params.albumId,
        newMediaItems: params.newMediaItems.map(item => ({
          description: item.description,
          simpleMediaItem: { uploadToken: item.uploadToken, fileName: item.fileName },
        })),
      },
      { headers },
    );
    return { data: response.data };
  } catch (error) {
    throw toError(error, 'mediaItems.batchCreate');
  }
},
```

New repository functions in src/api/repositories/photosRepository.ts:

```typescript
// Reads file, uploads bytes, calls batchCreate — atomic two-step
export async function uploadMedia(
  oauth2Client: OAuth2Client,
  filePath: string,
  mimeType: string,
  fileName: string,
  albumId?: string,
  description?: string,
): Promise<{ mediaItemId: string; uploadToken: string }>

// Separate lower-level batchCreate for callers that already have tokens
export async function batchCreateMediaItems(
  oauth2Client: OAuth2Client,
  newMediaItems: Array<{ uploadToken: string; fileName?: string; description?: string }>,
  albumId?: string,
): Promise<{ mediaItems: NewMediaItemResult[] }>
```

uploadMedia implementation notes:
- Use `import { readFile } from 'fs/promises'` to read file bytes as Buffer
- uploadToken from step 1 must be passed directly to step 2 — no storage between calls
- Call withRetry ONLY on the upload step (step 1). batchCreate failure after successful upload
  should NOT retry the upload (would create duplicate upload tokens). Call batchCreate without
  withRetry, handle its error separately.
- Return `{ mediaItemId: result.mediaItem?.id ?? '', uploadToken }` — mediaItemId from first
  result in newMediaItemResults array.

```typescript
    properties: {
      filePath: { type: 'string', description: 'Absolute path to the local file to upload' },
      mimeType: { type: 'string', description: 'MIME type (e.g., image/jpeg, video/mp4)' },
      fileName: { type: 'string', description: 'File name to use in Google Photos' },
      albumId: { type: 'string', description: 'Optional: album ID to add the media to immediately' },
      description: { type: 'string', description: 'Optional: description/caption for the media item' },
    },
    required: ['filePath', 'mimeType', 'fileName'],
  },
},

// Handler (note: uses appendonly scope — same PERMISSION_DENIED re-auth hint as create_album):
private async handleUploadMedia(request: CallToolRequest, tokens: TokenData) {
  const args = validateArgs(request.params.arguments, uploadMediaSchema);
  quotaManager.checkQuota(false);
  const oauth2Client = await this.getAuthenticatedClient(tokens);
  const result = await uploadMedia(oauth2Client, args.filePath, args.mimeType, args.fileName, args.albumId, args.description);
  quotaManager.recordRequest(false);
  return {
    content: [{ type: 'text', text: JSON.stringify(result, null, 2) }],
  };
}
```

CRITICAL anti-pattern to avoid: do NOT make upload and batchCreate a single withRetry call.
The upload step may succeed and return a token before batchCreate fails. Retrying the whole
thing would generate a new upload token on each attempt.
